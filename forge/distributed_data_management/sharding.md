# Sharding

## 1. Introduction
Sharding은 분산 데이터베이스에서 데이터를 여러 서버 또는 노드에 분할하여 주어진 resource로
데이터베이스의 horizontal-scaling(scale-out)을 가능하게 하는 데 사용되는 기술이다.
Sharding의 핵심은 대규모 데이터 세트를 더 작고 관리하기 쉬운 조각으로 나누고 여러 서버에 분산함으로써,
데이터를 더 빠르게 처리하고 액세스할 수 있으며 대량의 데이터를 처리할 때 발생할 수 있는 병목 현상 및 기타 문제를 방지할 수 있다.
즉, 시스템의 성능, availability 및 scalability를 향상시킬 수 있다.

### Definition of sharding
Sharding은 전체 dataset을 subset으로 나눠 shard로 저장하는 프로세스를 뜻하며,
각 shard는 다른 서버나 노드에서 관리할 수 있다. 데이터를 찾고 액세스하기 위해 sharding된 db는
데이터의 특정 특성 또는 속성을 기반으로 데이터를 특정 shard에 매핑하는 shard key를 사용한다.

### Explanation of why sharding is important
Sharding은 조직이 대량의 데이터를 보다 효율저긍로 처리하고 요구 사항이 변경됨에 따라 운영을 보다 효과적으로 확장할 수 있도록 지원한다.
또한 Sharding은 데이터가 단일 위치에 저장되지 않고 여러 서버에 분산되므로 안정성과 내결함성을 개선하는 데 도움될 수 있다.
이렇게 하면 하드웨어 장애 또는 기타 문제가 발생한 경우 데이터 손실 또는 손상을 방지할 수 있다.

## 2. How Sharding Works
Sharding은 대규모 데이터 세트를 샤드라고 하는 더 작고 관리하기 쉬운 조각으로 나누고
이러한 샤드를 분산 데이터베이스의 여러 서버 또는 노드에 분산시키는 방식으로 작동한다.

### Overview of the sharding process and its components
1. 샤딩은 큰 데이터 세트를 샤드라고 하는 더 작고 관리하기 쉬운 조각으로 분할한다.
   각 샤드는 일반적으로 데이터의 하위 집합을 포함하며 별도의 서버 또는 노드에 저장된다.
   효과적인 샤딩의 핵심은 성능과 확장성을 최적화하는 방식으로 데이터를 분할하는 것이다.
   이는 지리적 위치, 사용자 활동 또는 데이터 유형과 같은 다양한 요소를 기반으로 수행할 수 있다.
2. 데이터가 샤드로 분할되면 여러 서버 또는 노드에 분산됩니다.
   각 샤드는 일반적으로 redundancy와 availability을 보장하기 위해 여러 서버에 복제된다.
   즉, 한 서버에 장애가 발생해도 다른 서버에서 데이터에 계속 액세스할 수 있다.
   데이터를 분산하기 위해 샤딩 시스템은 Consistent Hashing 또는 range-based sharding과 같은
   다양한 기술을 사용하여 각 샤드가 서버 전체에 고르게 분산되도록 한다.
3. 샤딩된 시스템의 데이터에 액세스하는 것은 기존 데이터베이스보다 더 복잡할 수 있다.
   클라이언트가 데이터를 요청할 때 시스템은 먼저 데이터가 저장된 샤드를 결정해야 한다.
   여기에는 일반적으로 메타데이터 저장소 또는 인덱스를 쿼리하여 데이터 위치를 조회하는 작업이 포함된다.
   샤드가 식별되면 클라이언트는 적절한 서버에 쿼리하여 데이터를 검색할 수 있다.
   일부 샤딩 시스템은 캐싱 또는 로드 밸런싱을 사용하여 성능을 개선하고 데이터에 액세스하는 데 필요한 쿼리 수를 줄인다.

위와 같은 샤딩 프로세스에는 다음과 같은 몇 가지 주요 component가 포함된다.

- Shard key: 샤드 키는 데이터를 특정 샤드에 매핑하는 데 사용되는 고유 식별자이다.
  분할된 키는 위치, 사용자 ID 또는 타임스탬프와 같은 데이터의 다양한 특성 또는 속성을 기반으로 할 수 있다.
- Sharding algorithms: 샤딩 알고리즘은 데이터가 여러 샤드에 분산되는 방식을 결정한다.
  이는 지정된 값 범위를 기준으로 데이터를 나누는 range-based,
  또는 해시 함수의 출력을 기준으로 데이터를 나누는 hash-based 전략을 기반으로 할 수 있다.
- Shard metadata: 샤드 메타데이터에는 위치, 크기 및 상태와 같은 샤드에 대한 정보가 포함된다.
  이 정보는 쿼리를 라우팅하고 올바른 샤드에서 데이터에 액세스하는지 확인하는 데 사용된다.
- Query router: 쿼리 라우터는 샤드 키를 기반으로 올바른 샤드로 쿼리를 라우팅하는 역할을 한다.
  이를 통해 올바른 샤드에서 데이터에 액세스하고 쿼리를 최대한 효율적으로 처리할 수 있다.

Sharding은 database의 horizontal-scaling을 가능하게 하여 성능과 확장성을 개선하는 데 도움이 될 수 있으며,
데이터를 여러 서버에 분산시켜 안정성과 내결함성을 개선할 수 있다.
그러나 샤딩은 데이터 일관성 및 동기화, 로드 밸런싱 및 라우팅, 보안 및 개인 정보 보호 문제와 같은 새로운 문제를 야기할 수도 있다.

다음 섹션에서는 다양한 유형의 샤딩과 분산 데이터 관리에서 Sharding이 어떻게 사용되는지 살펴보자.

## 3. Types of sharding
분산 데이터 관리에 사용할 수 있는 여러 유형의 sharding strategies가 있으며 각각 고유한 장점과 단점이 있다.
가장 일반적인 유형의 샤딩은 다음과 같다.

### Range-based sharding
Range-based sharding은 데이터를 지정된 attribute 또는 key를 기반으로 contiguous range로 나누는 것이라고 할 수 있다.
예를 들어 데이터는 특정 시간 범위(예: 2019년에 생성된 모든 데이터)의 데이터를 포함하는 각 샤드와 함께 생성 시간을 기준으로 샤딩될 수 있다.
range-based sharding은 natural ordering이나 클러스터링이 있는 datasets에 효과적일 수 있지만,
데이터가 고르게 분산되지 않은 경우 구현하기 어려울 수 있다.  
다음은 지리를 기준으로 Range-based sharding한 예이다.

지리적 위치 정보가 포함된 고객 데이터 dataset가 있다고 가정.
특정 지역의 고객 데이터를 포함하는 각 샤드와 함께 이 데이터를 지역 또는 국가별로 샤딩할 수 있다.
예를 들어 하나의 샤드에는 유럽의 고객 데이터가 포함될 수 있고 다른 샤드에는 북미의 고객 데이터가 포함될 수 있다.
```rust
struct Customer {
    name: String,
    location: String,
}

fn shard_by_region(customers: Vec<Customer>) -> Result<Vec<Vec<String>>, String> {
    let mut shards: Vec<Vec<String>> = vec![vec![]; 2]; // create two shards

    for customer in customers {
        // shard customers by region
        match customer.location.as_str() {
            "Europe" => shards[0].push(customer.name),
            "North America" => shards[1].push(customer.name),
            _ => return Err(format!("Unknown region: {}", customer.location)),
        }
    }

    Ok(shards)
}
```

### Hash-based sharding
Hash-based sharding에는 해시 함수를 사용하여 함수의 출력을 기반으로 여러 샤드에 데이터를 배포하는 작업이 포함된다.
각 샤드에는 해시 값의 범위가 할당되고 샤드 키에 해시 함수를 적용한 결과를 기반으로 데이터가 샤드에 매핑된다.
이 접근 방식은 natural ordering이나 클러스터링이 없는 datasets에도 효과적일 수 있지만,
시간이 지남에 따라 데이터 분포가 변경되는 경우 구현하기 어려울 수 있다.  
다음은 소셜 미디어 플랫폼에서의 Hash-based sharding한 예이다.  
게시물의 해시태그 또는 키워드를 기반으로 이 데이터를 분할할 수 있다. 예를 들어 해시태그 `#food`가 포함된 모든 게시물은
하나의 샤드에 매핑될 수 있는 반면, 해시태그 `#travel`이 포함된 모든 게시물은 다른 샤드에 매핑될 수 있다.

```rust
use std::{
    collections::HashMap,
    hash::Hash,
};

struct Post {
    id: u64,
    text: String,
    hashtags: Vec<String>,
}

fn hash_shard(posts: Vec<Post>, num_shards: usize) -> Vec<Vec<Post>> {
    let mut shards: Vec<Vec<Post>> = vec![vec![]; num_shards];

    for post in posts {
        // calculate hash value for hashtags
        let mut hash_value = 0;
        for hashtag in post.hashtags {
            hash_value += hashtag.hash();
        }

        // map post to shard based on hash value
        let shard_idx = hash_value as usize % num_shards;
        shards[shard_idx].push(post);
    }

    shards
}
```

### Other types of sharding (e.g., list sharding, modulo sharding)
다른 유형의 sharding으로는 specified attribute 또는 키를 기반으로 데이터를 discrete lists로 나누는 list sharding과
modulo 작업을 기반으로 각 샤드에 고유 ID를 할당하는 modulo sharding이 포함된다.
이러한 유형의 샤딩은 특정 사용 사례에 효과적일 수 있지만 range-based 및 hash-based sharding보다 사용 빈도가 적다.
빈도는 적을 수 있지만 예를 들어 동적 샤딩을 구성한다고 가정해보자.  
샤드 수를 scaling 할때 키의 구성은 유지시키며 샤드를 확장해야할 필요가 있다면, 모듈로는 굉장히 매력적인 옵션이다.

sharding strategies를 선택할 때 distribution of data, dataset 크기 및 시스템의 성능 요구 사항과 같은 요소를 고려하는 것이 중요하다.
datasets가 발전하고 새로운 문제가 발생하고, 시간이 지남에 따라 샤딩 전략을 조정해야 할 수도 있다.

다음 섹션에서는 샤딩의 이점과 분산 데이터베이스의 확장성, 안정성 및 성능을 개선하는 데 어떻게 도움이 되는지 살펴보자.

## 4. Benefits of Sharding
샤딩은 분산 데이터 관리 시스템에 다음과 같은 몇 가지 이점을 제공한다.

### Improved scalability
샤딩은 대규모 데이터 세트를 더 작고 관리하기 쉬운 조각으로 나눔으로써 데이터베이스의 horizontal-scaling을 허용한다.
즉, dataset가 커짐에 따라 추가 노드 또는 서버를 추가하여 증가된 로드를 처리할 수 있다.
샤딩은 또한 데이터가 여러 서버에 더 고르게 분산되므로 더 나은 리소스 활용을 가능하게 한다.

### Increased availability, reliability and fault tolerance
샤딩은 분산 데이터베이스의 가용성, 내결함성과 안정성을 향상시킬 수 있다.
데이터가 여러 노드 또는 서버에 분산되어 있으므로 하나의 노드 또는 서버에 장애가 발생해도 반드시 데이터 손실이 발생하지는 않는다.
대신, 다른 사용 가능한 노드나 서버에서 데이터를 검색하여 시스템이 계속 작동하도록 할 수 있다.
또한 샤딩은 데이터가 각 서버나 노드에서 완전히 복제되는 대신 여러 샤드에 걸쳐 복제될 수 있으므로 보다 효율적인 데이터 복제를 허용할 수 있다.
이를 통해 단일 장애 지점의 영향을 줄이고 시스템의 전반적인 가용성과 안정성을 향상시킬 수 있다.

### Better performance and speed
샤딩은 또한 분산 데이터베이스의 성능과 속도를 향상시킬 수 있다.
여러 서버 또는 노드에 데이터를 분산함으로써 시스템은 더 많은 read/write 작업을 병렬로 처리하여 응답 시간을 줄이고
전체 시스템 성능을 향상시킬 수 있다. 또한 샤딩은 값비싼 하드웨어 업그레이드 또는 특수 인프라의 필요성을 줄여 조직의 비용을 절감할 수 있다.

전반적으로 샤딩은 분산 데이터베이스의 확장성, 가용성, 안정성 및 성능을 개선하여 크고 복잡한 데이터 세트를 처리하는 데 더 적합하도록 만들 수 있다.

## 5. Challenges of Sharding
샤딩은 상당한 이점을 제공할 수 있지만 시스템의 효율성과 안정성을 보장하기 위해 해결해야 하는 몇 가지 과제가 존재한다.

### Data consistency and synchronization
샤딩은 서로 다른 샤드 간에 데이터의 일관성 및 동기화를 유지하기 어렵게 할 수 있다.
데이터가 여러 노드에 분산되면 모든 노드에 동일한 최신 정보가 있는지 확인하기가 더 어려워진다.
데이터가 일관되지 않으면 데이터 손상, 손실 또는 보안 위반과 같은 심각한 문제가 발생할 수 있다.
이 문제에 대한 솔루션으로는 데이터 복제를 사용하거나 샤드 전체에 분산 트랜잭션을 구현하는 것이 포함될 수 있다.

### Load balancing and routing
샤딩은 워크로드가 모든 노드에 고르게 분산되도록 신중한 로드 밸런싱 및 라우팅이 필요하다.
특정 샤드가 너무 많은 트래픽을 수신하거나 노드에 과부하가 걸리면 응답 시간이 느려지거나 시스템 다운타임이 발생할 수 있다.
round-robin 또는 Consistent Hashing과 같은 로드 밸런싱 전략은 이 문제를 완화하는 데 도움이 될 수 있다.

### Security and privacy concerns
샤딩은 특히 multi-tenant 환경에서 보안 및 개인 정보 보호 문제를 일으킬 수 있다.
중요한 데이터가 여러 노드에 분산되어 있는 경우 무단 액세스 또는 breaches로부터 데이터를 보호하기가 더 어려워진다.
암호화, 액세스 제어 및 감사(auditing)와 같은 보안 조치는 이러한 문제를 해결하는 데 도움이 될 수 있다.

### Outline some of the trade-offs that come with sharding
샤딩은 많은 이점을 제공할 수 있지만 신중하게 고려해야 하는 trade-offs도 수반한다.
샤딩은 추가 인프라 및 관리 오버헤드가 필요하므로 시스템의 복잡성을 증가시킬 수 있다.
또한 샤딩은 데이터를 개별 파티션으로 분할해야 하므로 데이터 쿼리 및 분석 방법 측면에서 유연성을 감소시킬 수 있다.
샤딩의 이점과 장단점의 균형을 맞추는 것은 이 접근 방식을 구현할지 여부를 결정할 때 중요한 고려 사항이다.

### How Solana addresses these challenges
샤딩의 주요 과제 중 하나는 서로 다른 샤드 간에 데이터 일관성 및 동기화를 유지하는 것이다.
솔라나는 네트워크의 모든 트랜잭션에 대해 검증 가능하고 변경 불가능한 타임스탬프를 제공하는 역사 증명(PoH)이라는
고유한 합의 알고리즘을 사용하여 이 문제를 해결한다.
이를 통해 네트워크의 모든 노드가 트랜잭션 순서에 동의할 수 있으므로 샤드 전체에서 일관성을 유지하는 데 도움이 된다.

로드 밸런싱 및 라우팅도 샤딩에서 중요한 과제이다.
Solana는 각 샤드의 현재 로드를 기반으로 여러 샤드에 트랜잭션을 분산하는 동적 분산 로드 밸런서를 사용하여 이 문제를 해결한다.
이렇게 하면 단일 샤드가 과부하되지 않고 가능한 최대 처리량을 처리할 수 있다.

보안 및 개인 정보 보호 문제도 샤딩의 주요 과제이다.
솔라나는 합의 프로세스에 참여하기 전에 노드가 네트워크에 의해 검증되어야 하는 승인된 네트워크 모델을 사용하여 이러한 문제를 해결한다.
이를 통해 악의적인 행위자가 네트워크를 제어하는 것을 방지하고 신뢰할 수 있는 노드만 참여할 수 있도록 한다.

이러한 솔루션에도 불구하고 샤딩에는 여전히 복잡성 증가 및 유연성 감소와 같은 몇 가지 trade-offs가 있다.
Solana는 개발자가 특정 요구 사항에 맞게 네트워크를 customize할 수 있는 고도로 모듈화된 유연한 아키텍처를 제공하여 이러한 절충점을 해결한다.
이를 통해 유연성이나 유용성을 희생하지 않고도 샤딩의 이점을 실현할 수 있다.

다음 섹션에서는 샤드 키 선택, 데이터 배포 관리 및 데이터 일관성 유지를 위한 전략을 포함하여 샤딩에 대한 몇 가지 모범 사례를 살펴보자.

## 6. Best practices for sharding
샤딩은 대규모 데이터 세트를 관리하기 위한 강력한 도구가 될 수 있지만 시스템이 효율적이고 안정적이며 안전하도록 모범 사례를 따르는 것이 중요하다.

### Choosing the right sharding strategy for your use case
샤딩 전략을 결정할 때 데이터 분포, 쿼리 패턴 및 성능 요구 사항과 같은 요소를 고려하는 것이 중요하다.
Range-based Sharding은 natural ordering 또는 클러스터링이 있는 데이터 세트에 적합할 수 있는 반면,
Hash-based Sharding은 natural ordering이 없는 데이터 세트에 효과적일 수 있다.
고려해야 할 다른 요소로는 확장성, 데이터 일관성 및 보안이 있다.

- 적절한 샤드 키 선택: 샤드 키는 데이터 분포, 쿼리 패턴 및 확장 기능을 기반으로 선택해야 한다.
- 데이터 분포 관리: 단일 노드에 과부하가 걸리지 않도록 데이터가 모든 샤드에 고르게 분포되도록 하는 것이 중요하다.
- 데이터 일관성 유지: 샤드 간 일관성은 데이터 무결성과 안정성을 보장하는 데 중요하다.
  데이터 복제 또는 분산 트랜잭션을 구현하면 일관성을 유지하는 데 도움이 될 수 있다.
- 샤드 상태 및 성능 모니터링: 샤드 상태 및 성능을 모니터링하면 문제를 즉시 감지하고 해결하여 다운타임 또는 데이터 손실을 방지할 수 있다.
- 샤드 쿼리 최적화: 쿼리는 모든 노드에 고르게 분산되고 효율적으로 실행되도록 최적화되어야 한다.
- 샤드 마이그레이션 관리: 샤드 마이그레이션은 복잡하고 시간이 많이 소요될 수 있다.
  중단을 최소화하고 데이터 일관성을 보장하기 위해 샤드 마이그레이션 관리 계획을 세우는 것을 고려할 수 있다.

전반적으로 모범 사례를 따르면 샤딩이 효과적이고 효율적으로 구현되어 위험과 장단점을 최소화하면서 확장성,
안정성 및 성능의 이점을 제공하는 데 도움이 될 수 있다.

## 7. Real-world examples of sharding
샤딩은 많은 산업에서 크고 복잡한 dataset을 관리하는 데 널리 사용된다.
여기서는 샤딩을 성공적으로 구현한 회사의 실제 사례를 살펴보자.

### Case of companies that have successfully implemented sharding
Facebook: Facebook은 샤딩을 사용하여 수십억 명의 활성 사용자가 포함된 대규모 사용자 데이터베이스를 관리한다.
Range-based Sharding을 사용하여 데이터를 시간 범위 및 지리적 위치에 따라 더 작은 파티션으로 나눈다.

Uber: Uber는 샤딩을 사용하여 운전자의 위치 데이터와 승객의 승차 요청을 포함한 실시간 데이터 스트림을 관리한다.
Hash-based Sharding을 사용하여 여러 노드에 데이터를 분산하고 맞춤형 라우팅 및 로드 밸런싱 알고리즘을 사용하여 최적의 성능을 보장한다.

Twitter: Twitter는 샤딩을 사용하여 트윗 및 사용자 데이터의 방대한 datasets를 관리한다.
Range-based 및 Hash-based Sharding의 조합을 사용하여 여러 노드에 데이터를 분산하고 분산 캐싱 계층을 사용하여 성능을 향상시킨다.

### Commonalities of successful real-world examples
- 샤드 키 선택은 효율적인 데이터 배포 및 쿼리 성능에 중요하다.
- 가동 중지 시간과 데이터 손실을 방지하려면 샤드 상태와 성능을 모니터링하고 유지 관리하는 것이 중요하다.
- 로드 밸런싱 및 라우팅 전략을 사용하면 워크로드를 고르게 분산하는 데 도움이 될 수 있다.
- 샤드 환경에서 여러 샤드 간에 데이터 일관성을 유지하는 것은 어려울 수 있으며,
  이로 인해 데이터 손상이나 보안 위반과 같은 문제가 발생할 수 있다.
  데이터 무결성을 보장하려면 데이터 복제 또는 분산 트랜잭션과 같은 솔루션을 구현하는 것이 중요하다.
- 민감한 데이터를 여러 노드에 분할할 때 보안 및 개인 정보 보호 문제를 해결해야 한다.

성공적인 샤딩 구현의 실제 사례를 검토하여 자체 시스템에서 샤딩을 구현하는데 참고하자.

## 8. Sharding in Blockchain
Sharding을 사용하는 솔라나의 DB를 살펴보자.
솔라나는 고성능 분산형 블록체인 플랫폼으로 샤딩을 사용하여 높은 처리량과 낮은 트랜잭션 지연을 가능하게 한다.
Solana에서 샤딩은 "account sharding"이라는 기술을 사용하여 구현된다.

솔라나의 account sharding은 트랜잭션 정보가 포함된 계정을 더 작은 하위 집합으로 분할하고 이를
여러 노드에 분산시키는 작업을 포함한다. 솔라나 네트워크의 각 노드는 총 계정의 하위 집합을 관리할
책임이 있다. 이를 통해 Solana는 네트워크에 많은 수의 노드가 있는 경우에도 높은 트랜잭션 처리량과
낮은 대기 시간을 달성할 수 있다.

account 정보를 찾고 액세스하기 위해 Solana는 Pubkey와 해시 값의 조합을 사용한다.
Pubkey는 account를 식별하는 데 사용되며 해시 값은 account 관리를 담당하는 네트워크의 노드를 결정하는 데 사용된다.

솔라나의 account sharding은 확장성 향상, 자원 활용도 향상, 보안 강화 등 몇 가지 장점이 있다.
그러나 복잡성 증가 및 shard key의 신중한 관리 필요성과 같은 몇 가지 문제도 발생한다.

전반적으로 샤딩은 분산 데이터베이스의 sharding은 확장성을 향상시키는 기술이며,
솔라나에서 sharding을 사용하면 탈중앙화 네트워크에서 높은 트랜잭션 처리량과 짧은 대기 시간을 달성하는 데 효과 있음을 알 수 있다.

## 9. Conclusion
분산 데이터 관리 시스템의 샤딩 개념, 이점 및 과제, 이를 효과적으로 구현하기 위한 모범 사례를 살펴보았다.
또한 크고 복잡한 데이터 세트를 관리하기 위해 샤딩을 성공적으로 구현한 회사의 실제 사례 역시 살펴보았다.

샤딩은 확장성, 안정성 및 성능 측면에서 상당한 이점을 제공할 수 있지만 데이터 일관성, 로드 밸런싱 및 보안 문제와 같은 tradeoffs가 있다.
적절한 샤드 키 선택, 샤드 상태 및 성능 모니터링, 데이터 복제 또는 분산 트랜잭션 구현과 같은 모범 사례를 따르면
샤딩이 효과적이고 효율적으로 구현되어 위험과 장단점을 최소화하는 동시에 확장성과 안정성 이점을 제공할 수 있다.

Solana의 맥락에서 샤딩은 네트워크에서 처리되는 크고 복잡한 데이터 세트를 관리하기 위한 필수 도구이다.
Solana의 고유한 합의 알고리즘과 동적 분산 로드 밸런서는 샤딩과 관련된 많은 문제를 해결하므로
개발자가 확장 가능한 고성능 분산 응용 프로그램을 구축할 수 있는 강력한 도구가 된다.

결론적으로 샤딩은 분산 데이터 관리 시스템에서 크고 복잡한 데이터 세트를 관리하기 위한 중요한 기술이다.
효과적으로 구현하고 모범 사례를 따르면 샤딩은 확장성, 안정성 및 성능 측면에서 상당한 이점을 제공할 수 있다.
